# VulnAnalysisService Phase 1.5 — Smoke-Test-Driven Improvements

**Date:** 2026-06-18
**Status:** Approved
**Branch:** `feature/vuln-analysis-service` (fork-only)
**Predecessor:** `docs/superpowers/specs/2026-06-17-vuln-analysis-service-design.md`

## Evidence (live, `Application1.elf`)

PowerPC GHS firmware, 19,094 functions, 14 overlay spaces. Ran the SEVR
workflow end-to-end: `enumerate_attack_surface(depth=3)` → 16 reachable
functions → `detect_vuln_patterns(function=X)` per function. Two findings,
three observed limitations:

| Observation | Location |
| --- | --- |
| `format_string` FP: `err_str` is a phi of string literals; `reachesConstantOnly` returns false at `MULTIEQUAL` | `dm_input_task` @ `dnp_modbus.Initial::00016488` |
| Whole-program scan: 300 functions in 78s (≈260ms/fn, process-spawn dominated) | `detect_vuln_patterns` |
| Useful workflow is surface→scan, but it took a shell loop | manual |

## Changes

### 1. `reachesConstantOnly` recurses `MULTIEQUAL`

Add `case PcodeOp.MULTIEQUAL:` to the all-inputs-must-be-constant branch in
`PcodeQuery.reachesConstantOnly` (alongside `PTRSUB`/`PTRADD`/`INT_ADD`/
`INT_SUB`/`INT_MULT`). A phi of constants is constant. The existing seen-set
and step cap bound loop-carried phis. `definingOps`/`defChainHasInput` keep
their current MULTIEQUAL boundary semantics (unchanged).

### 2. Per-scan `DecompInterface` reuse

`detectVulnPatterns` (whole-program and attack-surface scopes) opens one
`DecompInterface` (`new DecompInterface(); openProgram(program); setOptions/
simplificationStyle` matching `FunctionService.decompileFunctionNoRetry`),
passes it to `scanFunction(program, f, active, decomp)`, and disposes in a
`finally`. `scanFunction` calls `decomp.decompileFunction(f, TIMEOUT_S,
TaskMonitor.DUMMY)` directly. The single-function path keeps the existing
`functionService.decompileFunctionNoRetry` (no reuse benefit for one call).
Scoped to one HTTP request — no cross-call lifecycle.

### 3. `scope="attack_surface"` mode on `detect_vuln_patterns`

New `@Param(value="scope", source=BODY, defaultValue="")`:
- `""` (default) → current behavior (whole program / single function).
- `"attack_surface"` → run the same BFS as `enumerateAttackSurface` (sources
  from `SinkCatalog`, callers-of via `ReferenceManager`), collect unique
  reachable functions, scan only those with the shared decompiler from #2.

New `@Param(value="max_depth", source=BODY, defaultValue="3")` clamped to
[0,8], used only when `scope="attack_surface"`. Response gains `scope` and
`attack_surface_function_count` when this mode is active. The BFS is
factored into a private `collectAttackSurface(program, maxDepth)` helper so
both `detectVulnPatterns(scope=attack_surface)` and `enumerateAttackSurface`
share one implementation.

## Testing

- Offline: `MULTIEQUAL(const,const)` → true, `MULTIEQUAL(const,param)` →
  false; existing 11 PcodeQuery tests unchanged. `scope="attack_surface"`
  test asserts only the BFS-reachable functions are scanned (mock harness
  reused from `enumerateAttackSurface_twoHopAndCycle`).
- Integration: re-run the live `Application1.elf` workflow with
  `detect_vuln_patterns(scope="attack_surface", max_depth=3)` — expect the
  `dm_input_task` format-string FP to disappear (#1) and the scan to complete
  in seconds, not minutes (#2).
