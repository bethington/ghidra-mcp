## NAMING CONVENTION ANALYSIS

### Consistent Patterns ‚úÖ

**Python MCP Tools**: `snake_case` (e.g., `get_function_variables`, `batch_create_labels`)

### Naming Mismatches Discovered During Analysis

During the initial comparison, several apparent discrepancies were found that are actually **naming convention variations**:

| Python Tool | Java Endpoint | Reason |
|-------------|--------------|--------|
| `get_xrefs_to` | `xrefs_to` | ‚úÖ Both valid - Python has "get_" prefix |
| `get_xrefs_from` | `xrefs_from` | ‚úÖ Both valid - Python has "get_" prefix |
| `get_function_labels` | `function_labels` | ‚úÖ Both valid - Python has "get_" prefix |
| `get_function_callees` | `function_callees` | ‚úÖ Both valid - Python has "get_" prefix |
| `get_function_callers` | `function_callers` | ‚úÖ Both valid - Python has "get_" prefix |
| `get_full_call_graph` | `full_call_graph` | ‚úÖ Both valid - Python has "get_" prefix |
| `get_function_call_graph` | `function_call_graph` | ‚úÖ Both valid - Python has "get_" prefix |

**Analysis**: The Python bridge adds a `get_` prefix to query endpoints for consistency with Python naming conventions (e.g., `get_function_variables`). The Java endpoints use the shorter form. Both are valid and intentional.
